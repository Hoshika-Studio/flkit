# Agent Instructions

This Flutter project was generated by FLKit. The project is intentionally
feature-first so humans and AI agents can extend it without guessing where code
belongs.

## Project Manifest

Read `flkit.yaml` before making architecture decisions. It records the FLKit
template, enabled tools, locales, and generated project conventions.

## Architecture

- Put feature code in `lib/features/<feature_name>/`.
- Keep feature UI in `presentation/`.
- Keep feature state, controllers, providers, or use cases in `application/`.
- Keep business models and contracts in `domain/`.
- Keep API clients, DTOs, repositories, and data mappers in `data/`.
- Keep feature translations in `i18n/` next to the feature.
- Keep shared app code in `lib/core/`.

## Internationalization

- Edit only `.i18n.json` files for translations.
- Do not edit generated Slang files in `lib/core/i18n/generated/`.
- Run `dart run slang` after changing translations.
- New feature translation files should follow the Slang namespace style, for
  example `userProfile_en.i18n.json`.

## Code Generation

- Run `dart run build_runner build --delete-conflicting-outputs` after changing
  Riverpod, Theme Tailor, Envied, Freezed, or JSON Serializable annotations.
- Do not edit generated `.g.dart`, `.freezed.dart`, or Theme Tailor output
  manually.

## Environment

- Environment values are generated with Envied from `.env`.
- Keep `.env.example` committed.
- Do not commit local secrets from `.env`.

## Validation

Before finishing changes, run:

```sh
dart run slang
dart run build_runner build --delete-conflicting-outputs
flutter analyze
flutter test
```
